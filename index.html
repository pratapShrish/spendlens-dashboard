    METHOD & HANDOFF

    How this works

    Currency conversion approach

    I keep exchange rates in one lookup object keyed by currency code. Because the supplied snapshot states a USD base, each rate represents how many units of that currency equal one USD. Every expense is normalized with USD = original amount ÷ rate. Summary cards, category totals, merchant rankings and the transaction table all call the same conversion function, so there is one source of truth and no duplicated arithmetic. Values stay at full precision during calculations and are rounded only for display, avoiding cumulative rounding errors.

    Adding a 25th currency would require only a new rate entry and making that code available in the form; the aggregation logic would not change. In a production version, the supported-currency list would be derived directly from validated rate data so those two steps could not drift apart.

    A null, missing, zero or non-numeric rate would otherwise produce an invalid or infinite USD value and corrupt every summary. The conversion function therefore rejects the value before division. This prototype prevents an unsupported currency from being submitted; a production system should also quarantine invalid imported rows, show a clear “rate unavailable” state, log the failure, and exclude those rows from totals until finance supplies a valid rate.
